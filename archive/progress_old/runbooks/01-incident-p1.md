# 런북 01 — P1 인시던트 대응

- **범위**: `docs/02 §9.3` 의 **P1** — 서비스 완전 장애 · 데이터 손실 위험 · 보안 침해 징후.
- **최초 대응 시간**: 알람 수신 후 **10 분 이내** 응답 (응답 = Slack 으로 수신 확인 + 담당자 지정).
- **문서 상태**: Foundation skeleton (세션 48). MVP 이후 실 인시던트 회고를 반영해 개정.

---

## 0. 언제 이 런북을 여는가

아래 중 하나라도 해당하면 **즉시 P1**.

- 프로덕션 API 완전 무응답(5xx > 50% 5 분 지속) — `docs/02 §9.3` 큐 길이 / 5xx 알람.
- 데이터 유실 가능성 있는 DB 오류 (권한 박탈 / drop 감지 / 마이그레이션 롤백 불가).
- 자격증명 노출 의심 (레포 secret leak, 키 유출 보고, 비정상 과금).
- 생성된 아바타가 **서명 검증 실패** 상태로 대량 배포된 정황.

애매하면 **P1 로 개시하고 triage 단계에서 강등**. 처음부터 P2 로 시작해 놓쳐 P1 으로 에스컬레이트하는 비용이 더 크다.

---

## 1. Detect (탐지)

**입력**: PagerDuty / Grafana / Slack `#alerts` / 사용자 신고.

1. 알람 원문을 **Slack `#incident` 로 크로스포스트** (화면 한곳으로 모음).
2. **인시던트 채널 생성**: `#inc-YYYYMMDD-<slug>` (예: `#inc-20260419-worker-503`).
3. 담당자 3 역할 지정 (겸임 가능):
   - **IC (Incident Commander)**: 의사결정. 기술 질문은 IC 가 차단.
   - **Ops**: 시스템 조작 (롤백 · 스케일링 · 트래픽 드레인).
   - **Comms**: 외부 커뮤니케이션 (상태 페이지 · 고객 Slack · 이메일).

**Done 기준**: 채널 생성 + 3 역할 지정 + 첫 알람 시각 기록.

---

## 2. Triage (분류)

**목표**: 5 분 이내 영향 범위 확정.

체크리스트:

- [ ] **무엇이** 망가졌는가: API? 워커? DB? 외부 벤더? — Grafana 대시보드 3종(`job-health`, `cost`, `quality`) 를 훑는다.
- [ ] **얼마나** 영향 받는가: 전체 / 특정 리전 / 특정 테넌트.
- [ ] **언제부터** 망가졌는가: 알람 시각 ≠ 최초 발생 시각. 로그 / 메트릭 타임라인으로 역추적.
- [ ] **최근 변경**: `git log --since="6 hours ago"` + 최근 배포 이력. **Foundation 단계에서는 대부분 이것이 원인**.
- [ ] **데이터 손상 여부**: 유실된 레코드 / 부분 쓰기 / 순서 뒤집힘 징후.

**강등 조건**: 5xx < 10% · 단일 테넌트만 영향 · 롤백 없이 복구 가능 → **P2 로 강등** 후 본 런북 종료.

**Done 기준**: 영향 범위 5 문장으로 `#inc-*` 채널에 요약.

---

## 3. Mitigate (완화)

원칙: **복구 우선, 원인 분석은 나중.**

후보 액션 (상황에 따라 선택):

1. **배포 롤백** — 가장 흔한 복구. `kubectl rollout undo deployment/<name>` 또는 Argo/Helm 에서 이전 revision 으로. `docs/02 §10` 멱등성 덕분에 재실행 안전.
2. **트래픽 드레인** — Ingress 레벨에서 영향 경로만 차단 (다른 경로 살림).
3. **의심 어댑터 비활성화** — `infra/adapters/adapters.json` 의 해당 엔트리 `enabled: false` 후 재배포. `routeWithFallback` 이 다음 후보로 자동 내려감.
4. **키 회전 (보안 침해 시)** — 노출 의심 API key / 서명 키 즉시 회전 + 레지스트리(`infra/registry/signer-keys.json`) 에서 해당 `key_id` `status: "revoked"` 로 전환 + 신규 발급 키로 재서명.
5. **DB 롤백** — 마이그레이션 실패 시. **반드시 IC 결재 후** 진행 (데이터 손실 재확인).

각 조치 후 **1 분 관찰** — 메트릭이 정말 돌아왔는지 확인 후 다음 조치 또는 소통 단계로.

**Done 기준**: P1 에서 P2/P3 로 강등 가능한 상태 (= 사용자 경험 회복) 달성.

---

## 4. Communicate (소통)

### 4.1 첫 공지 (탐지 후 15 분 이내)

**필수 3 요소**:

> **[탐지]** 2026-04-19 10:34 UTC — Geny 2D Avatar API 의 아바타 생성 엔드포인트 5xx 급증을 탐지했습니다.
> **[영향]** 전체 사용자의 아바타 생성 시도. 기존 번들 다운로드 / 프리뷰는 영향 없음.
> **[다음 업데이트]** 10:50 UTC 또는 상태 변화 즉시.

### 4.2 후속 공지 (15~30 분 간격)

진행 상황 / 가설 / ETA 를 간결히. 가설이 뒤집히면 그 사실을 명시 — 침묵보다 정정이 낫다.

### 4.3 복구 공지

> **[복구]** 2026-04-19 11:12 UTC — 해결되었습니다. 원인은 배포 <commit-sha> 의 회귀였고, 롤백으로 복구했습니다. 상세 RCA 는 48 시간 내 공유합니다.

**Done 기준**: 상태 페이지 "Resolved" + `#inc-*` 채널에 복구 공지 pin.

---

## 5. Postmortem (회고)

- **작성자**: IC.
- **기한**: 24h 내 draft → 1주 내 확정.
- **위치**: `progress/postmortems/YYYY-MM-DD-<slug>.md`.
- **원칙**: blameless. 개인 귀책이 아니라 **시스템이 그 실수를 허용한 이유** 에 집중.
- **필수 섹션**: 타임라인 · 영향 / 사용자수 / 금전적 임팩트 · 근본 원인 · 재발 방지 액션(담당자 + 기한) · 잘한 점(keep).

**Done 기준**: 재발 방지 액션이 트래커(세션 백로그 또는 이슈) 에 등록되고 1주 내 착수 가능한 상태.

## 부록 A — 역할 치트시트

| 역할 | 하는 것 | 하지 않는 것 |
|---|---|---|
| IC | 의사결정 · 역할 분배 · 외부 소통 게이트 | 직접 `kubectl` / `psql` — 검증 분리 |
| Ops | 시스템 조작 · 메트릭 해석 · IC 에게 옵션 제시 | 소통 · 고객 응대 |
| Comms | 상태 페이지 · Slack · 이메일 · 고객 재질문 흡수 | 시스템 조작 |

## 부록 B — 자주 쓰는 링크

- Grafana: `infra/observability/grafana/dashboards/` 3종.
- Prometheus alerts: `infra/observability/prometheus/alerts.yaml`.
- Alertmanager routing: `infra/helm/observability/values.yaml` (P1→PagerDuty / P2→Slack).
- 서명 키 레지스트리: `infra/registry/signer-keys.json`.

## 부록 C — 훈련(Drill)

- 분기마다 **game day** — 가짜 알람으로 런북 실행, 소요 시간 측정.
- 측정 지표: MTTD(탐지까지 시간) · MTTA(응답까지 시간) · MTTR(복구까지 시간). 목표 MTTA < 10 분 / MTTR < 1 시간 (Foundation 기준, MVP 에서 재설정).
