## 결정 (D1~D3)

### D1 — physics.json 은 왜 자동 이식하지 않았는가
`physics/physics.json` 의 9→12 setting 확장은 **구조 추가 + 물리 튜닝** 두 축을 동시에
건드린다. 구조 추가만 보면 기계적이지만, 각 setting 의:
- Normalization 축 (X vs Y vs angle).
- Input weight 분배 (head_angle_x 70w / head_angle_y 30w / body_angle_x 20w 등).
- Output scale × weight 곱 (accessory_back scale=1 w=80 / accessory_front scale=0.8 w=70).
- Vertex radius 체인 (radius 5, mobility 1.0, delay 0.55).
- `meta.total_input_count` / `total_output_count` / `vertex_count` 재계산.

전부 **물리 튜닝 결정**이다. 저자가 보지 않고 자동 덮어쓰기 하면 기존 저작 튜닝을 파괴할
위험이 크다. 따라서 migrator 는 touching 하지 않고, TODO 로 한 줄 남긴다 — "이것이
유일한 저자 개입 지점" 이라고 명시.

**Why:** 물리 튜닝은 자동화 범위 밖. 기존 저작 훼손 방지가 우선.
**How to apply:** 차후 rig 버전 bump 시 동일 기준 — 구조 추가(mechanical) vs 튜닝 결정
(authoring) 을 명확히 분리하고, 후자만 TODO 로 남긴다.

### D2 — `writeIfAbsent` / `appendIfMissing` 로 idempotent 보장
저자가 이미 `ahoge.spec.json` 을 수동 작성했거나, 동일 outDir 에 migrator 를 재실행하는
경우가 있을 수 있다. 기존 파일을 덮어쓰면 저작 수정이 사라진다.
- `writeIfAbsent`: 파일 없을 때만 쓰기. 저자 작성본 보존.
- `appendIfMissing`: `## 6. v1.3.0 추가분` 센티넬이 이미 있으면 skip.
- `deformers.json` 삽입은 이미 `existingIds.has(n.id)` 로 중복 방지 구조.
- `parts/accessory_*.spec.json` 의 parent 이동은 `=== "accessories_layer"` 가드로
  이미 이동된 경우 no-op.

결과: case C (determinism) 가 여전히 pass. 동일 src 2회 migrate → 동일 결과.

### D3 — `deformers.json` 필드명은 `nodes`
초기 디자인 시 `deformers` 로 이름이 혼동되기 쉬우나, 실제 스키마는 `nodes` 를 사용.
`d.nodes.map(...)` 로 접근. 이를 잘못 쓰면 즉시 TypeError (`map of undefined`) 발생해
테스트에서 드러남. 새 패치를 만들 때는 `schema/v1/deformers.schema.json` 을 먼저
확인할 것.
