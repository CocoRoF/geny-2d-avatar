## 1. 목표

halfbody rig 템플릿을 v1.0 → v1.1 → v1.2 에서 **v1.3.0 으로 한 단 더 올리는 마이그레이터**
를 자동화한다. 실 PhysicsSetting 저작(9 → 12)과 `ahoge.spec.json`/`deformers.json` 본문은
세션 10 D1 "보수적 마이그레이션" 원칙에 따라 `MIGRATION_REPORT.md` TODO 로 남기고, 이번
세션은 "스키마 + 파라미터 + cubism_mapping" 만 기계적으로 확장한다.

근거: 세션 10 migrator 가 이미 v1.0→v1.2 까지 같은 규약으로 짰고, 부가 스펙/물리 기하 생성은
수공예(Cubism 에디터 흐름) 영역이라 자동화 금지. 대신 체인이 누락될 리스크 자체를 migrator
체인 + golden 으로 막는다.

```
rig-templates/base/halfbody/v1.0.0 ──▶ v1.1.0 ──▶ v1.2.0 ──▶ v1.3.0
                                              (세션 10)       (세션 27, this)
```

## 3. 설계 결정 (D1–D5)

### D1. "기계적 3 아이템만" — 물리 저작은 TODO 로 남김

v1.3.0 이 실제 production 에 쓰이려면 PhysicsSetting 9 → 12 로 확장된 **실제 geometry
(bone/joint link/weight/damping)** 가 필요. 이는 Live2D Cubism Editor UI 작업이므로
마이그레이터가 만들어낼 수 없다. 기계적으로 확장 가능한 3 가지만 자동화:

1. `template.manifest.json` 의 `version` bump + `cubism_mapping` 3 엔트리 추가
2. `parameters.json` 에 3 신규 파라미터 object append
3. `MIGRATION_REPORT.md` TODO 로 나머지 저작 작업 리스팅

근거: 세션 10 D1 "migrator 는 보수적" 과 동일 원칙. 파이프라인이 자동 생성한 허위 물리
setting 을 goldenset 이 통과시키면 오히려 버그가 숨음.

### D2. 3 신규 파라미터는 모두 `channel: "extension"` + `physics_output: true`

`ahoge_sway` / `accessory_back_sway` / `accessory_front_sway` 는 모두 "물리엔진이 자동 계산해
Cubism 에 되돌려주는 출력" 역할. `channel: "extension"` 으로 분리해두면 web-avatar runtime 이
"유저 제어 + 감정 + 물리 출력" 3 채널을 각자 스케줄링할 때 겹침 없이 처리 가능.

근거: docs/05 §3 ChannelMixer 설계가 `extension` 채널을 "non-user, non-emotion output" 으로
정의함. 세션 10 `cloth_main_fuwa` 와 같은 channel.

### D3. 체인 체계 유지 — v1.0.0 에서도 v1.3.0 으로 직진 가능

`MIGRATORS` 배열은 인접 hop 만 정의하지만, `migrate.mjs` 러너가 `from === currentVersion`
발견 시 순차 적용. 따라서 v1.0.0 소스에서도 `node migrate.mjs` 한 번에 1.0→1.1→1.2→1.3
전체 체인이 진행. test case A 가 이를 보장.

근거: 이미 세션 10 에서 검증된 체인 러너. 신규 hop 은 "끝에 push" 만으로 자동 연결.

### D4. 결정론 테스트 (case C) — golden 이 아닌 "두 번 돌려 같은가"

파일 하나로 frozen sha 를 박제하는 대신, 같은 소스를 두 번 migrate 해 JSON 문자열이 완전
일치하는지 확인. 이유는:

- migrator 출력은 tmp 디렉터리에 매번 새로 찍힘 → 디렉터리 경로 포함 sha 는 의미 없음
- 순서·객체 키 순서가 흔들리면 case A/B 가 실패 → case C 가 안전망
- goldens 디렉터리(`rig-templates/base/halfbody/v1.3.0/`)는 이번 세션에서 생성하지 않음
  (D1 원칙 — 실 물리 저작 이전에는 "디렉터리만" 생성하면 낭비)

근거: 세션 26 post-processing 골든 sha 와 다른 전략. "외부 바이너리 없는" 회귀로 동일한
효과.

### D5. `MIGRATION_REPORT.md` TODO 를 4 건으로 쪼갬 — 단일 "TODO v1.3.0" 이 아님

각 TODO 는 담당 파일이 다르고 담당 역할도 다르므로(이트리 저작 vs. JSON 편집 vs. 수식
매핑) 분리해 두어야 릴리즈 직전 체크리스트 로 역할. migrator 자체는 pass 하지만 "production
ready" 플래그는 이 TODO 4 건 소진 + goldenset 1 회 통과 로 정의.

근거: 세션 10 D3 "migrator 와 Release Gate 는 분리" 를 그대로 계승.

## 5. 위험·후속

- **실 PhysicsSetting 부재**: v1.3.0 디렉터리 자체를 아직 저장소에 만들지 않음. 외부 rig 소비자가
  "v1.3.0 은 언제 나오냐" 물을 때 "migrator 는 세션 27, 저작은 세션 30+" 로 답할 수 있도록
  INDEX 에 명시.
- **goldenset 14 종 미확장**: v1.3.0 rig 실 번들 회귀는 `exporter-core` goldens 가 아직
  v1.0/v1.1/v1.2 만 커버. 실 저작 후 goldens 재생성 필요(세션 10 D4 `regen-goldens.mjs`
  재활용).
- **체인 누락 감지**: `MIGRATORS` 배열에 hop 을 빼먹으면 조용히 실패 가능. test case A 가
  "1.0 → 1.3 전체 체인 manifest.version" 확인으로 safety net 역할.
