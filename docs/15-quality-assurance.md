# 15. 품질 보증 (Quality Assurance)

> **한 줄 요약**: AI가 만든 결과물은 본질적으로 불확실하다. QA는 **"불확실성을 지표로 바꾸고, 게이트로 막고, 데이터로 개선하는"** 시스템이다. 수동 검수는 항상 필요하지만, 자동으로 걸러낼 수 있는 것은 전부 자동으로.

---

## 1. QA 원칙

1. **Quality is a pipeline stage, not a phase.** QA는 릴리스 전 아니라 **매 파이프라인 단계 안에** 있다.
2. **Measure, don’t feel.** 주관 단어("자연스럽지 않다")를 지표로 번역한다.
3. **Shift-left**: 개발자가 PR 단계에서 먼저 차단.
4. **Shift-right**: 프로덕션 트래픽에서도 지속 관측.
5. **Golden set is law.** 골든셋 회귀는 배포의 절대 기준.
6. **Document known failures, not just successes.** 실패 사례 카탈로그가 자산.

---

## 2. QA 계층 (Layers of Quality)

```
[L5] Post-release: 실 사용자 오버라이드·신고·재생성 시그널
[L4] Shadow mode:   새 모델/알고리즘의 병행 평가
[L3] Canary:        1~5% 트래픽 선행 배포
[L2] Staging:       인수 테스트(UAT), 골든셋 회귀
[L1] CI PR:         단위/통합 테스트, 정적 분석
[L0] Dev local:     pre-commit, quick sanity
```

각 계층은 위 계층으로 넘길 기준(Exit criteria)을 갖는다.

---

## 3. 자동 테스트 (Automated Tests)

### 3.1 단위(Unit)

- 각 알고리즘(알파 정리, 색 이동, 선 두께 정규화)을 **고정 입력 → 수치 출력** 테스트.
- 프롬프트 빌더, 스키마 검증, 권한 체크.
- 목표: 핵심 모듈 라인 커버리지 ≥ 80%.

### 3.2 통합(Integration)

- 파이프라인 노드 간 계약(Schema) 테스트.
- Mock AI 어댑터로 DAG 1회 완주.
- DB 마이그레이션, 이벤트 브로커 퍼블리시/서브스크라이브.

### 3.3 E2E(End-to-End)

- Playwright: 사용자 시나리오 4종(홈→생성→프리뷰→export).
- 테스트 전용 플래그로 AI 콜을 **캐시/결정론 모드**로 고정.

### 3.4 성능(Performance)

- k6 / Locust: 생성 50 동시 요청, 수치 SLO 측정.
- 에디터 번들 사이즈 체크(>80KB gzip 실패).

### 3.5 접근성(A11y)

- Axe automated scan + 수동 키보드/스크린리더 체크리스트.

---

## 4. 골든셋(Golden Set)

### 4.1 구성

- 사내 합성 아바타 100대.
- 다양성: 체형 5, 스타일 10, 템플릿 5, 의상/악세 변형 다수.
- 라이선스: 전량 내부 소유.

### 4.2 운영

- 주 1회 전체 회귀 실행, PR 당 중요 20대 서브셋 실행.
- 지표: 검수 점수, LPIPS 변화, 파츠 경계 ΔE, 알파 노이즈.
- 실패 시: 원인 라벨을 붙여 이슈로 자동 생성.

### 4.3 유지

- 분기 1회 신규 아바타 10대 추가, 오래된 10대 아카이브.
- 사용자 데이터는 절대 포함하지 않는다.

---

## 5. 품질 지표 사전(Quality Metric Dictionary)

| 지표 | 정의 | 계산 | 사용처 |
|---|---|---|---|
| **ValScore** | 검수 렌더러 점수 0–100 | 08 §6 | 게이트, 대시보드 |
| **ΔE_mean_ctx** | 색 문맥 내 평균 ΔE | Lab 변환 후 평균 편차 | 후처리 Stage 3 |
| **L_dir_err** | 광원 방향 오차 | deg | Stage 4 |
| **Seam_noise** | 경계 2px 링 ΔE 최대 | 95p | Stage 5 |
| **FaceID_sim** | 원본과 얼굴 유사도 | 0–1 | 검수 |
| **CLIP_style_d** | 스타일 프로파일 거리 | 0–1 | 검수 |
| **Break_count** | 포즈 세트 깨짐 픽셀 | 개 | 자동 적합 |
| **AutoPass_rate** | 자동 통과율 | % | 제품 품질 |
| **Override_rate** | 사용자 강제 통과율 | % | 경고 신호 |
| **Regen_rate** | 파츠 재생성 빈도 | % | UX/품질 |

## 6. 릴리스 게이트(Release Gates)

### 6.1 파이프라인 레벨

- [ ] 골든셋 `ValScore` 평균 변화 ≤ 1.0, 개별 ≤ 3.0.
- [ ] 시각 회귀(LPIPS) p95 변화 ≤ 0.08.
- [ ] 성능 SLO 회귀 없음.

### 6.2 플랫폼 레벨

- [ ] 보안 스캔: P0/P1 0.
- [ ] 에러 예산(Error budget): SLA 대비 여유 확인.
- [ ] Migration dry-run 통과.

### 6.3 제품 레벨

- [ ] i18n 문자열 누락 0.
- [ ] 접근성 체크리스트 통과.
- [ ] 문서 업데이트 여부 확인.

---

## 7. 사람 검수 시스템 (Human Review)

### 7.1 사용자 셀프 리뷰

- UI에서 파츠·포즈별 검사, 재생성/수동 보정.
- 이슈 태그를 사용자 선택 가능(우리에게 신호).

### 7.2 내부 QA (사내)

- α/β 초기에 내부 아트 QA 팀이 **샘플 20대/주** 정기 검수.
- 체크리스트 기반 점수화.
- 자동 점수와 사람 점수의 **상관** 을 측정해 자동 점수 가중치 재조정.

### 7.3 커뮤니티 리뷰 (GA)

- 마켓 프리셋에 대해 사용자 리뷰(별점/후기).

### 7.4 QA 도구

- 전용 대시보드: 아바타 카드 그리드, 원클릭 렌더 재생, 태그 붙이기.
- 원본/이전 버전 비교 슬라이더.

---

## 8. A/B 테스팅 & 실험

### 8.1 실험 플랫폼

- 내부 feature flag + 실험 시스템 (예: GrowthBook 또는 자체).
- 실험별 성공 지표·최소 표본 크기·중단 기준 사전 정의.

### 8.2 대표 실험 후보

- 프롬프트 빌더 A vs B: 파츠당 재생성율.
- 검수 게이트 임계 78 vs 82: 사용자 오버라이드율.
- 기본 스타일 프리셋 순서: 온보딩 완주율.

### 8.3 주의

- 품질 지표 악화 시 **자동 롤백**.
- 데이터 파편화: 세그먼트별 결과 구분 분석.

---

## 9. 지속 품질 관측(Production QA)

### 9.1 대시보드

- **Quality Health**: ValScore 분포, override 비율, 재생성 분포.
- **Per-template**: 템플릿별 품질 드리프트.
- **Per-vendor**: 어댑터별 실패/ΔE.

### 9.2 알람

- AutoPass_rate 주간 -10%p → P2.
- Override_rate 일간 > 25% → P2.
- Vendor failure 비율 > 20% → 자동 라우팅 감축.

### 9.3 드리프트 탐지

- 스타일 임베딩 클러스터 변화.
- 지역/언어 세그먼트별 지표 차이.

---

## 10. 회귀 & 이슈 관리 (Issues)

### 10.1 태깅

- 실패 케이스는 자동 태깅: `face_misaligned`, `iris_out_of_sclera`, `hair_clip`.
- 사용자 리포트도 동일 태그 체계.

### 10.2 트리아지

- 주간 리뷰 회의: 상위 10 태그, 상위 10 사용자 불만.
- 액션: 파이프라인 수정, 프롬프트 개선, 템플릿 개선, UX 개선.

### 10.3 재발 방지

- 수정된 케이스는 골든셋/회귀 세트에 편입.
- 재발 시 자동 테스트가 먼저 잡도록.

---

## 11. 데이터 품질 (Data QA)

### 11.1 업로드 품질

- 해상도/포맷/내용 검사.
- 초상권·저작권 메타 확인(자기 소유 선언).

### 11.2 스타일 프로파일 품질

- 임베딩 분포 정상성.
- 프로파일과 결과물 간 거리 분포.

### 11.3 스키마 품질

- 필수 필드 결측률, enum 이탈, 타임스탬프 역전.

---

## 12. 프롬프트 품질 관리(Prompt QA)

### 12.1 프롬프트 템플릿 리뷰

- 새로운 프롬프트 템플릿은 PR.
- **정성 비교**: 같은 아바타 집합에 신/구 템플릿 적용 → 차이 렌더.
- **정량**: 재생성율, ValScore.

### 12.2 위험 프롬프트 탐지

- 자유 텍스트의 안전 필터.
- 금칙 카테고리(폭력/성적/타인 초상).
- 우회 패턴 탐지(연쇄 대체어).

---

## 13. 보안 품질(Security QA)

- 정적 분석, 의존성 스캔(13 §8).
- 업로드된 파일에 대한 포맷·콘텐츠 검사.
- 스토리지 권한 경계: 프리사인 URL scope, 교차 org 접근 거부 테스트.
- 침투 테스트(외부) 연 1회.

---

## 14. 성능 품질(Perf QA)

- 에디터 FPS ≥ 50 (프리뷰 재생).
- 파이프라인 SLO (02 §12).
- Export 시간 SLO (11).
- 부하 테스트: 동시 아바타 생성 100 → p95 이상치 없음.

---

## 15. 배포 전략(Rollout)

### 15.1 Canary

- 새 파이프라인 노드 버전은 1% → 5% → 25% → 100% 단계별.
- 각 단계에서 ValScore 비교.

### 15.2 Shadow

- 새 AI 어댑터는 shadow로 **결과만 수집**(사용자 미노출) 2주.

### 15.3 Auto-Rollback

- canary 가 기준 미달 10분 → 자동 롤백.

---

## 16. 장애 대응(Incident Response)

### 16.1 심각도

- **P0**: 서비스 전체 중단, 데이터 손실 위험.
- **P1**: 주요 기능 불가.
- **P2**: 일부 기능 저하.
- **P3**: 미세 이슈.

### 16.2 SLA (내부)

- P0 감지 → 응답 5분, 대응 30분.
- P1 → 응답 15분, 대응 2시간.
- P2 → 영업시간 내 대응.

### 16.3 포스트모템

- P0/P1 은 48시간 내 무비난(blameless) 포스트모템.
- 액션 아이템 추적, 재발 방지 자동 테스트 편입.

---

## 17. QA 도구 & 인프라

- **Allure** 또는 유사 — 테스트 리포트.
- **Renovate/Dependabot** — 의존성 최신화.
- **Chromatic / Percy** — 시각 회귀(프런트 UI).
- **LPIPS/SSIM** — 렌더 회귀.
- **Sentry** — 에러 모니터.

---

## 18. QA 문화

- "품질" 은 QA 팀 혼자의 일이 아니다. 모든 엔지니어가 테스트를 쓴다.
- 리뷰어는 **테스트 없는 PR** 을 반려 가능.
- 1년 1회 "Quality Week": 부채 청산/회귀/문서화.

---

## 19. 열린 질문

- 골든셋의 사용자 다양성 확보 방법(내부 합성만으로 부족할 가능성).
- "자동 점수 vs 사람 점수" 상관 향상을 위한 학습 기반 랭커를 도입할지.
- A/B 실험 플랫폼 자체 구축 vs 외부 SaaS.
